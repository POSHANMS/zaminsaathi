# ZaminSaathi

A three-agent AI system helping rural Karnataka families understand their land documents and plan next steps.

## Project Status

### ✅ Built and Working
- **frontend/index.html** — React single-file frontend UI.
- **frontend/serve.py** — Python HTTP server serving the frontend locally.
- **agents/parser_agent.py** — Agent 1 (Parser): Reads any land document JSON, auto-detects type (RTC/MUTATION/SURVEY), loads the correct skill, and returns a plain-English explanation of every field.
- **agents/detector_agent.py** — Agent 2 (Detector): Cross-checks documents against each other AND the MCP database; runs security permission check before acting and writes to the audit log after; catches survey number mismatches.
- **agents/planner_agent.py** — Agent 3 (Planner): Takes detector output and generates a concrete action plan with office names, portal URLs, documents to carry, and officer script.
- **mcp_server/server.py** — FastMCP server with `query_land_record` tool querying `mcp_server/data/land_records.json`.
- **mcp_server/data/land_records.json** — Synthetic Karnataka land records for demo (3 records).
- **security/permissions.py** — Access control layer: checks permissions before Agent 2 acts.
- **security/audit_log.py** — Audit log layer: appends flags to `audit_log.jsonl` after Agent 2 acts.
- **skills/rtc-skill/SKILL.md** — RTC document skill context.
- **skills/mutation-skill/SKILL.md** — Mutation document skill context.
- **skills/survey-skill/SKILL.md** — Survey Sketch skill context.
- **sample_docs/** — 4 synthetic Karnataka land documents (rtc_001, mutation_001, survey_001, rtc_002) with a planted survey number mismatch (142/3 vs 142/4) that Agent 2 detects.
- **audit_log.jsonl** — Append-only audit log (auto-generated by Agent 2 runs).
- **.env** — GEMINI_API_KEY loaded via python-dotenv.
- **.gitignore** — Excludes .env, __pycache__, audit_log.jsonl.

### 🔲 Still To Build
- **backend/app.py (or app.py)** — Flask backend that wires the three agents to a POST /api/analyse endpoint.
- **docker/docker-compose.yml** — Docker Compose config to start all services with one command.

## Tech Stack
- **Agent Framework:** Google ADK (google-adk 2.3.0)
- **LLM:** Gemini 2.5 Flash (via Gemini Developer API)
- **MCP Server:** Python FastMCP
- **Security:** Two separate layers — permissions.py (before) + audit_log.py (after)
- **Skills:** Demand-loaded only when document type detected
- **Frontend:** React (CDN, single HTML file)
- **Backend:** Flask
- **Deployment:** Docker Compose (pending)

## How to Run Agents (Dev Mode)
```bash
# Set API key
export GEMINI_API_KEY=<your_key>   # Windows: $env:GEMINI_API_KEY="<your_key>"

# Run Parser Agent test
$env:PYTHONIOENCODING="utf-8"; python agents/parser_agent.py

# Run Detector Agent test (runs parser first, then detector)
$env:PYTHONIOENCODING="utf-8"; python agents/detector_agent.py

# Run Planner Agent test (runs parser, detector, then planner)
$env:PYTHONIOENCODING="utf-8"; python agents/planner_agent.py
```

## Running the Frontend
To view and run the interactive frontend demo without browser security blocks (CORS/CDN restriction blocks on `file://` urls), start the local Python server:
```bash
python frontend/serve.py
```
Then open your browser and navigate to:
[http://localhost:3000](http://localhost:3000)

## Key Design Rules (from AGENTS.md)
1. Never give legal advice — always appends disclaimer
2. Never store real user documents — demo uses synthetic data only
3. Security has TWO separate layers — never mixed:
   - Permission check = BEFORE Agent 2 acts
   - Audit log = AFTER Agent 2 acts
4. Never hardcode API keys
5. Agent Skills load on demand only
6. Kannada output only if manually verified — default is English
